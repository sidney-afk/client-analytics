# ROLLBACK.md — the "back to a working website in one step" runbook

**This file is law.** The owner's requirement, verbatim in spirit: *"I need to make sure that if
there's a bug or a problem or anything happening, I can click on a button and everything goes
back to normal. Our whole business depends on this."* Every phase of the independence plan is
executed under the rules below, and the **Live State table** in this file must be updated in the
same PR as any change it describes. If this file is out of date, the migration is out of
compliance — stop and fix it.

**Meaning of “one step.”** A kill flag must contain new harmful behavior in one step. It does not
prove that queued writes are safe to drain or that authority can be reversed without classification.
Track B's full team rollback remains blocked by F27 until an audited per-team
quarantine/classify/replay/discard path exists; use `docs/ops/FLIP_RUNBOOK.md`, never a default
drainer, for that recovery.

For the coverage map across sync edges, nightly checks, alerts, and containment/recovery paths, see
[`docs/ops/MONITORING.md`](docs/ops/MONITORING.md).

A successful pager or relay execution proves only the observed happy path. It does not prove retry
behavior, lane isolation, terminal Slack delivery, B1 cursor safety, or monitoring independence
(F09/F66/F81/F131/F132). Repair-list and linkage alerts are not currently immediate: live logic
requires two distinct nonzero summaries and shares one throttle across all V2 nonzero classes.
Treat immediate behavior as a target until F132 closes. Do not remove the independent Samples GitHub
schedule while F132 is open; if reducing duplicate cadence, remove the pager's last-branch dispatch.

## 1. Standing rules (non-negotiable)

1. **One-step behavior rollback, always.** Every cutover ships behind a server-readable runtime
   kill switch. Flip that switch first; a frontend revert is secondary because stale tabs and Pages
   propagation are not immediate. `git revert` restores source, not necessarily the same Edge
   runtime: F51 proved floating dependencies/latest CLI/no artifact attestation. No EF rollback may
   claim “exact” until a pinned source-closure/toolchain manifest, release SHA, independent server
   fingerprint, and a supported exact-platform-artifact restore procedure are captured and drilled.
   A Pages deploy/revert or update-banner check likewise does not expire stale callers (F127).
   Mandatory rollback releases require an embedded build/auth-authority epoch, server rejection
   before mutation, privacy-safe population proof, and draft/queue-safe reload plus fresh identity.
2. **The old path stays available until the new path has baked.** n8n workflows being replaced are
   normally left ACTIVE during canary, then only DEACTIVATED (never deleted) after their gate passes,
   then archived only at final cleanup. Activation is a rollback only when the exact approved graph,
   trigger, credential, error, and last-green contract has been read back. **Explicit exception:**
   `MJbMZ789B5ExZz9x` is inactive after a crash cluster and must not be blindly activated (F46). Before
   touching ANY n8n workflow, export its JSON — **raw workflow JSON contains hardcoded keys and
   must NEVER be committed to this public repo (rule 8 wins): export to the private weekly-backup
   Drive folder and commit only a public-safe status stub to `n8n-backups/` (dated), the
   Phase-0/A2 precedent** — in the same PR as the change.
3. **Additive-only database changes during the entire migration.** New tables and new columns
   are allowed; DROP/RENAME/type-changes are forbidden until the final cleanup phase after
   everything has baked. This guarantees the old code path always still works against the
   current schema — which is what makes rule 1 honest.
4. **Snapshot before every phase.** (a) git tag `pre-<phase>` (e.g. `pre-A1`) on `main`;
   (b) JSON export of every n8n workflow the phase touches; (c) Supabase dump (CSV or SQL) of
   every table the phase writes; (d) for every EF, a source-closure/dependency/JWT/toolchain/release
   manifest plus independent downloaded server fingerprint, and—once the platform-supported route
   is discovered—the exact deploy artifact needed for restore. Private artifacts stay in the
   private backup store; only public-safe names/hashes go in `EXECUTION_LOG.md`.
5. **Log everything.** `EXECUTION_LOG.md` (create in the first execution PR) gets a dated entry
   for every deploy, flag flip, n8n change, DB migration, backup taken, incident, and rollback —
   with enough detail to reconstruct events later. This complements the in-app event ledgers
   (owner decision D7).
6. **Gates are hard stops.** At every gate in `docs/independence/INDEPENDENCE_PLAN.md` §6: post the evidence
   (test results, canary metrics, reconciler-correction count), get the owner's explicit OK,
   only then proceed.
7. **Verify the rollback, not just the deploy.** For each phase, before its canary starts,
   actually EXERCISE the recovery once on the QA client. For Track B this means FLIP_RUNBOOK R2's
   stop/snapshot/classify/resolve/team-zero sequence before any authority reversal—not a direct
   flip-back. Re-soak before flipping forward. An untested recovery is a hope, not a button.
8. **The repo is public.** No secrets in code, docs, commits, or logs — ever.
