# SyncView Website + Automation Audit - 2026-07-03

Scope: full code and automation orientation for `client-analytics`, covering the
public SyncView website, Supabase functions, GitHub Actions, and the connected
n8n instance. No production code or workflow changes were made during this
audit.

Live site: `https://syncview.synchrosocial.com`
Repository: `sidney-afk/client-analytics`
n8n instance: `synchrosocial.app.n8n.cloud`

---

## Executive Summary

SyncView is a working internal operations system that has grown beyond a simple
static dashboard. The core logic is more mature than a prototype: there are
calendar/sample conflict guards, live Supabase reads, extensive regression
tests, dedicated nightly browser tests, and a backup workflow.

The main risk is architectural. The app is served as a public static website,
while many important writes and business actions are performed by public n8n
webhooks. The browser UI currently acts as the first layer of access control,
but anything inside `index.html` is inspectable by anyone.

The recommended path is not a large rewrite. The practical path is phased:
harden high-impact webhooks first, attach alerting everywhere, fail closed on
client links, review Supabase policies, then gradually split the single-file app
into safer modules.

---

## Current Architecture

- `index.html` is the application: about 35,136 lines and roughly 2.45 million
  characters.
- GitHub Pages serves the site from `main`.
- Supabase is used for browser-side reads with a committed publishable key.
  README says anonymous reads are permitted and writes go through n8n.
- n8n is the write and integration layer for Supabase, Google Sheets, Linear,
  Slack, HubSpot, Stripe, Gmail, eSignatures, TikTok, AI services, and backups.
- Google Sheets remains a human-readable mirror, and for analytics data it is
  still described as the source of truth.
- The connected n8n instance currently has 87 workflows, 33 credentials, and 3
  data tables visible through the MCP connection.

## What Is Strong

- The repo has unusually detailed regression coverage for a single-file app.
  The unit suite exercises many subtle calendar/review race conditions.
- Live browser tests are separated from push/PR checks because they intentionally
  write to production-like backend paths for the test client.
- The calendar and sample-review upsert workflows include real defensive logic:
  phantom-row protection, stale-save conflict checks, link-clobber protection,
  duplicate-link protection, and comment merge protection.
- Supabase migration notes show good instincts for sensitive TikTok pilot
  tables: RLS enabled, no browser-readable policy, and explicit revoke rules.
- The client-credentials Edge Function has a staff passphrase check and does not
  rely only on browser visibility.
- A weekly backup workflow exists and is active.
- An n8n error-alert workflow exists and is intended to DM Sidney when workflows
  fail.

## Priority Findings

### P0 - Staff Access Is Mostly a Browser-Side Gate

Evidence:

- `index.html:25762` defines `_AUTH_KEY='syncview_auth_v1'`.
- `index.html:25765` compares the typed password to a literal in the browser.
- `index.html:25769` stores the pass state in `localStorage`.

Impact:

This is useful against accidental access, but it is not durable access control.
Because the repo and static site are public, the browser gate can be inspected
and bypassed.

Recommendation:

Put staff-only surfaces behind real authentication. Practical options include
Supabase Auth, Google Workspace auth, Cloudflare Access, or a small server/Edge
session layer. The existing password overlay can remain temporarily as a visual
gate while real auth is introduced behind it.

### P0 - Public Webhooks Perform Important Writes and Actions

Evidence:

- `index.html` hardcodes many n8n webhook URLs.
- Inspected critical workflows report "No credentials required for this
  webhook."
- Examples include calendar writes, sample-review writes, Linear status changes,
  Linear comments, onboarding submission, templates, Slack notifications, and
  sales intake.

Impact:

If a webhook performs a write or calls a privileged service, hiding the button in
the UI is not enough. A crafted request can go directly to n8n unless the
workflow validates the caller.

Recommendation:

Add request verification to write/action webhooks. Prioritize staff-only actions
and business workflows first, then client-facing review actions.

### P0 - Sales, Stripe, and Contract Workflows Need Stronger Verification

Evidence:

- `?intake=1` bypasses the staff password and opens the sales intake flow.
- `index.html:31416` posts to `/webhook/sales-intake-submit`.
- The Sales Intake workflow can create an agreement, write to Supabase, and send
  a combined signing/payment email.
- The Stripe invoice workflow uses a normal webhook shape; provider signature
  verification was not visible in the inspected workflow.
- The contract-signed workflow checks a literal value inside workflow logic.

Impact:

These automations affect real customer communication, CRM stages, agreements,
and onboarding. They should not depend on the obscurity of a URL.

Recommendation:

Add provider-signature verification for Stripe and eSignatures events before any
downstream action. Put the sales intake submit path behind staff auth or a
signed intake session.

### P0 - API Access Values Are Embedded in n8n Code Nodes

Evidence:

- Active Linear workflows include API access values directly inside Code node
  JavaScript.
- The same pattern is mentioned in older repo audit notes.

Impact:

Embedding sensitive API access in workflow code increases exposure through
workflow exports, screenshots, backups, and future edits.

Recommendation:

Move those values into managed n8n credentials or environment-backed config.
Keep workflow code free of raw API access strings.

### P1 - Client Links Fail Open When the Review Key Is Missing

Evidence:

- Client link checks appear around `index.html:25403`, `index.html:25446`,
  `index.html:25485`, and `index.html:25668`.
- When a client has no review key configured, the app warns in the console and
  continues.

Impact:

Any client missing its review key can be opened with only a client query
parameter.

Recommendation:

Populate every client review key, then make missing keys fail closed. If a grace
period is needed, keep it explicit and temporary.

### P1 - Credential History Stores Password Values

Evidence:

- `supabase/functions/client-credentials/index.ts:225` includes `password` in
  `DIFF_FIELDS`.
- `old_value` and `new_value` are written into credential event rows.

Impact:

The event table can become a second password store, including historical
passwords. That increases blast radius for backups and future data access.

Recommendation:

Redact password values in event history. Store that a password changed, who
changed it, and when, but not the previous or new password contents.

### P1 - Supabase Browser Reads Need Table-by-Table Review

Evidence:

- README says the browser uses a committed publishable key and anonymous SELECT.
- Several modules read Supabase REST directly from the browser.

Impact:

This is acceptable only when RLS policies are exact. If anonymous reads can see
more rows than intended, client-side checks do not provide true data isolation.

Recommendation:

Audit every table reachable by the browser role. For client-facing views, move
toward scoped reads through Edge Functions or tighter per-client RLS.

### P1 - n8n Has Recent Failure Concentration

Evidence:

- Recent execution search found 585 error executions since 2026-07-01.
- Many errors were concentrated in `Sample Review - Upsert`.
- Several recent errors also came from `SyncView Calendar - Upsert Post`.
- The error-alert workflow exists, but sampled workflow settings did not show it
  attached everywhere.

Impact:

High-volume workflow errors can look like random app failures unless production
workflows alert and error rates are reviewed.

Recommendation:

Attach the error-alert workflow to all production workflows. Create a simple
workflow-health summary grouped by workflow ID and status. Investigate the
sample-review burst first.

## Medium Findings

### The Single-File App Is Now a Maintenance Risk

Measured facts:

- `index.html` has about 35,136 lines.
- There are 277 `innerHTML =` assignments.
- There are 473 inline `onclick=` occurrences.
- There are 232 `localStorage` occurrences.

This is still workable, but it makes every future edit more expensive. Routing,
state, rendering, API calls, and feature modules share one global surface.

Recommendation:

Do not rewrite the whole app first. Extract in safe layers: API clients, auth
helpers, render/escape helpers, then feature modules.

### Inline HTML and Event Handlers Increase Review Cost

The app has good escaping helpers, including `_calEsc`, `_calEscAttr`, and
`_jsAttrArg`, which is a positive sign. The risk comes from volume: many UI
paths are string-built and event-handler based.

Recommendation:

For new work, prefer safer rendering patterns. After inline handlers are reduced,
add a Content Security Policy.

### Production Deploys Directly From `main`

README says GitHub Pages serves `index.html` from `main`. A bad merge can ship
immediately.

Recommendation:

Add a staging or preview path. Protect `main` with required unit tests. Use the
live E2E suite before large workflow-affecting changes.

### n8n Workflow Hygiene Needs Structure

There are 87 workflows and no tags. Some active workflows include `TEST` in the
name, some have no description, and one active workflow description says it is
inactive until a credential is attached.

Recommendation:

Tag workflows by area and status: `prod`, `test`, `sales`, `calendar`,
`samples`, `onboarding`, `backup`, and `deprecated`. Require every production
workflow to have an owner, description, alerting setting, and backup note.

### Backup Inventory Is Split

The weekly backup workflow is active and useful. The repo's `n8n-backups/` folder
is partial/historical compared with the live 87-workflow inventory.

Recommendation:

Keep the Drive backup, but add a generated repo inventory: workflow name, ID,
active flag, endpoint, owner, criticality, and last backup timestamp.

### Local Tests Are Not Fully Windows-Safe

Verification:

- `npm.cmd test` ran 34 unit suites.
- 32 passed.
- 2 failed locally:
  - `test/overnight-runner-output-path.js`
  - `test/overnight-runner-singleton-lock.js`
- Both call `spawnSync('bash', ['-lc', ...])`.
- `where.exe bash` did not find bash on this Windows machine.

Recommendation:

Either skip those checks on Windows with a clear message, use `bash.exe` when
Git Bash is installed, or rewrite those checks in Node. CI runs Ubuntu, so this
looks like a local developer experience gap rather than a production regression.

## Rollout Guidance

- Do not rewrite the whole app first.
- Snapshot any n8n workflow before editing it.
- Add access checks to one high-risk webhook at a time.
- Keep old routes working behind temporary compatibility paths where needed.
- Add tests for each tightened gate before enforcing fail-closed behavior.
- Use staging/preview for auth and webhook changes before shipping to `main`.

## Bottom Line

SyncView is a substantial internal operations product with meaningful safeguards
already in place. The next improvement phase should focus on trust boundaries:
real staff auth, verified automation entry points, clearer n8n ownership,
workflow health visibility, and a gradual move away from one large browser file.
