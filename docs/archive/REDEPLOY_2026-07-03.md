# Redeploy note — 2026-07-03

The GitHub Pages deployment for the merge commit of PR #662
(`86e395adb8fe04ebd55f885d41bdbb1be6a20e6d`) got stuck in a failed state on
GitHub's backend: every deploy attempt — including a full rebuild that produced a
fresh artifact — failed at the `Getting Pages deployment status…` step with the
generic `Deployment failed, try again later.`

Diagnosis at the time (all healthy, ruling out a real problem):

- `build` job green; the failure was only in the infrastructure `deploy` step,
  which uploads an already-built artifact and never parses site content.
- DNS for `syncview.synchrosocial.com` resolved correctly to GitHub Pages, and
  the live site served `HTTP/2 200` over a valid HTTPS cert throughout.
- `CNAME` and the other Pages control files were correct and unchanged.
- githubstatus.com reported GitHub Pages and Actions fully operational.

The Pages deployment record is keyed on the commit SHA, so retrying the same
commit kept colliding with the wedged record. This commit exists only to advance
`main` to a new SHA, which mints a fresh Pages deployment record and clears the
stuck deploy. No application code is changed.
