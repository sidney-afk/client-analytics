## 4. Issue structure (sampled: Henry Ammar, Jenna Phillips Ballard, Eben and Annie, Jesse Israel, Sidney Laruel TEST)
**Pattern confirmed: parent issue = batch, sub-issue = deliverable.**
- Batches are created as **mirrored parents, one per team**: e.g. "Henry Ammar · 11 Jun 2026" exists as GRA-6335 AND VID-12636 (identical description). Batch title formats seen: `{Client} · {11 Jun 2026}`, `{CLIENT} | July 1st`, `{CLIENT} | March 2nd - March 6th | REELS`, `{CLIENT} | SAMPLES | VID-GRA`, `{CLIENT} | Graphic Samples` / `| Video Samples` / `| YouTube Samples`.
- **Parent carries**: filming-plan Google Doc link, Google Drive raw-footage folder, per-video timestamp mapping ("Video #3 = Video #4 on filming plan, PART 1 34:00-58:00"), editing instructions; sample-batch parents carry the creative brief with inline images hosted at `uploads.linear.app/{orgId}/...` (signed JWT URLs). Parent usually assigned to the lead editor; parent state loosely maintained (In Progress / For SMM approval / Todo).
- **Sub-issues**: titles "Video 1..12", "Sample 01..03", "REEL #N", "Thumbnail #N", "{Client} | Thumbnail #N". VID sub description template: `**Main Camera:** \n**Side Camera:** \n**Audio:**`; GRA sub description = the hook/title text (e.g. "She Needs To Feel You"); older REEL subs embed Drive file links + timecode ranges + notes. Created seconds apart → scripted creation (n8n/SyncView), createdBy = the SMM (raha, ludmila, camila) or sidney.
- **Cross-team parenting exists**: GRA-6095..97 (thumbnails) are children of VID-12105; ALE-245..250 children of VID-7621. Also **issues move teams and keep alias identifiers**: querying VID-12400 returns GRA-6217 (Jesse Israel thumbnail moved VID→GRA) — a replacement sync must handle identifier aliasing.

**Fields actually used on sub-issues**: `state` (the core signal), `assignee` (always set = editor/designer), `dueDate` (nearly always set; see nightly-bump automation below), `project`, `parentId`, `cycle` (auto-assigned weekly). **NOT used**: labels (empty on every sampled current issue), estimate, priority (0/None on all current work; legacy issues have values), attachments (empty everywhere sampled), milestones, relations, SLA fields.

**Comments are load-bearing** (two uses):
1. Editor/designer posts the **deliverable link as a comment** (e.g. Rocío Perez commenting a `drive.google.com/file/...` link on GRA-6402). Deliverables are NOT Linear attachments.
2. SyncView writes review feedback via the Linear API **as sidney@synchrosocial.com** with body prefix `**Kasper (via SyncView):**\n\n{feedback}` (seen on VID-12642, GRA-6402: "tweaks", "Low quality", "Feels a little too edited...").

**Automation fingerprints observed in Linear data**:
- Nightly due-date roller: dozens of issues have `updatedAt` at ~23:00:xx UTC daily with `dueDate` moved to the next day (overdue items perpetually bumped) — clearly an n8n cron.
- Bulk cycle rollover / reconcile touches: e.g. all Eben & Annie actives share `updatedAt 2026-05-19T14:18`.
- SyncView QA probes in TEST project "Sidney Laruel" (`137d80cc`): issues "test syncview probe — graphic/video" (GRA-6444, VID-12715, self-canceled after validating link-paste), "Sample Review Test — (parent)" GRA-6495/VID-12727 with Sample sub-issues; bulk-archived 2026-07-02T16:32.
