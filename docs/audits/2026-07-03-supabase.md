## 2. RLS & grants — what the anon key can do

**Anon key can NEVER write anything.** No table has any INSERT/UPDATE/DELETE policy for anon/authenticated. All writes go through service_role (n8n credential `XdBpJ6Xk8PMpZXXT`, or the two Edge Functions' injected service key), which bypasses RLS. Verified live per docs: anon INSERT → 401 (AUDIT_2026-06-15.md, SAMPLES_SUPABASE_KICKOFF.md).

| calendar_posts | ✅ policy `"anon read calendar_posts"` `using(true)` (SQL in CALENDAR_REALTIME_MIGRATION.md; live policy DDL not in repo) | ❌ | any client can read any client's rows |
| content_samples | ✅ `using(true)` | ❌ | |

## 6. Schema gaps vs the two goals

### (a) n8n → Edge Function writes: what n8n enforces that the DB does not
The DB enforces almost nothing; every one of these lives only in n8n Code nodes (or FE) and must be ported or pushed into the DB:
1. **Column allow-lists** (`ALLOWED` arrays per writer) — the DB has no column-level protection; service role can write any column. Calendar ALLOWED (2026-06-14 backup) = 29 cols; later columns (`thumb_rev`, `kasper_finished_at/closed_at`, `kasper_finish_log`, `title_*`) were added to the live list per the rollout docs but no post-06-14 backup captures it. sample_reviews ALLOWED captured in `sample-review-upsert.2026-06-25.initial.json`.
2. **The guard gauntlet** (calendar-upsert-post): read-failure guard (store outage ≠ "row not found"), phantom-row guard (no CREATE with zero CONTENT_FIELDS), `__CLEAR_LINK__` sentinel (plain `''` never clears a Linear link), link-clobber guard (carry stored link forward when incoming blank), duplicate-link guard (no two live rows sharing one `linear_issue_id` — NOT a DB unique constraint), conflict/LWW window (`comments_base_at` vs stored `updated_at` — not a DB compare-and-swap).
3. **No status vocabulary constraints** — `status`, `video_status`, `graphic_status`, `caption_status`, `approval` etc. are unconstrained text everywhere except client_credentials/client_credential_events (which have CHECKs). Nothing stops an invalid status at the DB layer.
4. **`updated_at` stamping** is n8n code (`new Date().toISOString()` as text) except inside the merge RPCs; no DB default/trigger.
5. **Overall-status recompute + stale-approval clearing** (`computeOverallStatus`, `_calClearStaleApprovals`) — FE + reconciler, never DB.

8. Already DB-side and reusable as-is: the two merge RPCs (atomicity) and the two `*_status_at` triggers.

### (b) Linear replacement: what exists vs what's missing
**Exists today (Linear-mirroring assets):**
- `workload_issues` — the only real Linear mirror: per-sub-issue status/status_type, assignee (id/name/email), due_date, parent (`parent_id`/`parent_identifier` = the "batch"), team, client_name, url, active flag. But it is a documented throwaway cache keyed by Linear's issue id, rebuilt every 10 min, with denormalized text parents and no FK to cards.
- Card↔Linear link columns: `calendar_posts.linear_issue_id`/`graphic_linear_issue_id` (+ same on `sample_reviews`), with `video_status_at`/`graphic_status_at` change-stamps and the whole reconcile apparatus (2 scripts, 2 GH Actions, 2 n8n triggers, ledgers, CAP=15).
- `sample_review_events` already models `linear_in`/`linear_out`/`reconcile` sources — a ready-made pattern for an in-app activity feed.
- Status mapping Linear-state→card-status exists in 3 synchronized copies (FE `_calMapLinearStatusStrict`/`CAL_PRIORITY`, n8n, reconciler-extracted).

**Missing for in-app editor/designer views (no table exists for any of these):**
1. **Batches** — Linear parent issues have no first-class table; `workload_issues.parent_id/parent_identifier` is the only trace. Need a `batches` (or `deliverable_batches`) table: id, client_slug, name, due window, status, created_by.
2. **Deliverables as own entities** — today a deliverable is either a calendar card component or a sample component; there is no production-side entity carrying editor-facing state (brief, revision round, files) independent of the review card. Either extend `calendar_posts`/`sample_reviews` or add a `deliverables` table FK'd to cards + batches.
3. **People/roster** — no users/team_members table anywhere. Editor identity lives in Linear (`assignee_*` on the mirror) and in Google Sheets ("Video Editors" tab with Slack mapping used by `send-urgent-slack`; SMM map via `_kasperLoadSMMMap()`). Need `team_members` (id, name, email, role editor|designer|smm|kasper, slack_user_id, active).
4. **Assignment** — no assignee column on `calendar_posts` or `sample_reviews`; assignment exists only inside Linear. Need `assignee_id` per component (video/graphic) or per deliverable, plus due_date per component (`scheduled_date` is the publish date, not the production due date; due_date exists only on the Linear mirror).
5. **Auth + per-user RLS** — there are zero Supabase auth users; identity is self-declared names in localStorage + one shared staff passphrase. Editor views that WRITE require real auth (Supabase Auth) and per-role/per-client policies; every existing policy is anon `using(true)` read-only. This is also the repo's own standing Phase-4 debt.
6. **Status vocabulary for production states** — Linear workflow states (per-team, e.g. "For SMM Approval", "Tweak Needed") would need a home: either CHECK/enum on new status columns or a small `statuses` lookup, replacing the 3-copy mapStatus machinery.
7. **Clients table** — none; `client` is a bare slug text column everywhere and client metadata (display name, review tokens) lives in Google Sheets. A Linear-projects replacement ("projects = clients") implies a `clients` table.
8. **Id minting** — Linear issue ids/identifiers (VID-/GRA-) are used as join keys (`workload_issues.id`, link columns, dedupe-by-Linear-issue in FE); removing Linear removes that namespace; deliverables need native ids and the duplicate-link guard becomes a real unique constraint.
9. Type debt that a rebuild should fix: text timestamps (except `*_status_at`), text `order_index`, JSON-as-text comment threads, no FKs anywhere, no NOT NULLs.
