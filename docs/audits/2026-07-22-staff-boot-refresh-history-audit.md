## Executive verdict

The staff app is not yet boot-perfect. This audit registers **14 new findings:
1 BLOCKER, 12 IMPORTANT, and 1 NICE-TO-HAVE**. It also adds a controlled
password-handoff observation to existing F155 rather than assigning a duplicate
number.

The blocker is a protected-state lifetime defect: a rendered weekly-report
surface is not part of the shared staff-identity purge and has no BFCache-return
revalidation owner, so already-rendered protected content can remain visible
after identity loss. The most common correctness pattern below it is
latest-request ownership: Production, Workload, weekly filters, the first-party
uploader, and the gated admin workspace can all retain or publish work from the
wrong route, filter, document lifetime, or client selection.

No finding below authorizes a fix. The owner questions define the choices that
must be answered before remediation is split into small route-specific PRs.

## Owner questions

Each line is a decision boundary, not an implied answer.

- **OQ-R2-01 / F186:** Should identity loss immediately replace every protected staff surface with a sign-in gate, including on persisted BFCache return?
- **OQ-R2-02 / F155:** After password acceptance, should the opaque gate remain until verification settles, or hand off immediately to a destination-specific neutral skeleton?
- **OQ-R2-03 / F192:** After an uploader queue refresh fails, should the page retain labelled last-good rows or replace the queue with a blocking error and Retry?
- **OQ-R2-04 / F193:** Should Upload wait for account mappings, or paint the form immediately with a mapping-specific loader until readiness is known?
- **OQ-R2-05 / F187:** Should omitted Production parameters always restore canonical defaults: all teams, active issues, and project details open?
- **OQ-R2-06 / F188:** Should Production revoke reads on route/pagehide and enforce one latest-request-wins generation on return?
- **OQ-R2-07 / F190:** Must a persisted Workload return begin a new freshness generation, or is one polling interval of stale state acceptable?
- **OQ-R2-08 / F189:** Should reload paint cached Workload and immediately revalidate its watermark, or bypass the cache on explicit reload?
- **OQ-R2-09 / F191:** On weekly-report refresh failure, should the route retain labelled last-good data or replace it with a blocking error?
- **OQ-R2-10 / F197:** Should unknown notification counts remain visibly pending/error instead of disappearing as zero?
- **OQ-R2-11 / F198:** Which More-menu surfaces must force-refresh on BFCache return, and which may retain clearly labelled last-good state?
- **OQ-R2-12 / F194:** Should the first-party uploader lock one client per compose session, or support switching through abortable per-client leases?
- **OQ-R2-13 / F195:** May caption generation use the default after a prompt-read failure, or must it fail closed until the client prompt is confirmed?
- **OQ-R2-14 / F196:** Is draft continuity between staff Submit and public intake intentional, or must their storage namespaces be isolated?
- **OQ-R2-15 / F199:** Is local deduplication sufficient, or should “Saved” require a typed server receipt and later reconciliation?
