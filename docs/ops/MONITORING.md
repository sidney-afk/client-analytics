contain or roll it back. A one-step kill may stop new behavior; F27 and F51 prove that safe team
authority recovery and exact Edge Function rollback require additional evidence.

## Known monitoring and evidence gaps — go-live blockers

These are not covered by the green watchers above. A green pager or reconciler therefore cannot
waive them.

| Finding | Missing detector / evidence | Required closure |
|---|---|---|
| F47 — urgent-review delivery truth | No recipient-specific pending-age or receipt check catches a staff/SMM/Kasper urgent action that resolves no exact recipient while the UI reports **Sent**. | Persist resolved-recipient and delivery outcome, show an honest unresolved state, and alert when recipient-specific pending work exceeds the owner-approved SLA. |
| F48 — Editors-week exposure/completeness | No monitor proves the unauthenticated response is access-contained or complete across pagination; native parity also misses transitions and historical attribution. | Contain the endpoint first; then add role-scoped, fully paged parity assertions and page on proven omissions or foreign rows. |
| F49 — backups and capacity | Repository tests do not prove the latest backup exists or restores, and no documented alarm covers database/egress headroom, spend control, or plan limits. | Capture Dashboard evidence, complete a timed restore drill, record thresholds, and route capacity/backup-staleness alerts to an owned destination. |
| F50 — deliverable/card state divergence | Reconciler linkage counts do not detect a changed Production status whose linked Calendar/Samples review status remains stale. | Define the mapping or canonical reader, then add a cross-surface mismatch metric and TEST scenario before either team flips. |
| F51 — deployed-function provenance | A green deploy job does not attest every deployed Edge Function's exact source/artifact, commit, secret-independent configuration, or supported rollback object. | Emit and retain an all-function release manifest; independently fingerprint/read back every deployed function; prove one exact-artifact restore. |
| F52 — provider credential containment | A plaintext credential remains in a reachable workflow branch and retained versions, while complete consumer/access/scope/provider-usage evidence is absent. | Restrict access immediately; stage and TEST-prove the managed replacement for the known live branch; owner revoke/rotate; then finish the broad census while monitoring unknown-consumer failures. |
| F53 — Graphics artifact lifecycle | No monitor proves a designer attached/replaced the canonical deliverable file and that SMM/client surfaces see the same artifact. | Implement the protected artifact contract, then add assignment-to-delivery and replacement-visibility probes with audit evidence. |
| F54 — inactive-client work | Queue counters and write guards do not exclude inactive clients, so apparently healthy totals may include offboarded work. | Filter every queue reader and reject writes server-side; alert on any inactive-client row that is renderable, assigned, or mutated. |
| F55 — authority vocabulary split | No cross-consumer check detects a legacy `supabase` side that the browser rejects while backends normalize it to SyncView authority. | Constrain/migrate flag writes, remove the alias, and run an all-consumer contract probe; page and freeze forward changes on any invalid/partial/disagreeing parse. |
| F58 — lane-specific emergency stop | A normal-mode `off` readback does not prove the independent parity lane is disabled, and current summaries do not give an operator one explicit “no lane can write” verdict. | Expose/read both F2/F4 decisions and active-lane counts; drill the dual kill; page if an incident stop leaves either lane writable. |
| F60 — teardown restore proof | Monitoring shows runtime health, not whether every retired webhook/workflow/credential has an exact captured inverse that was successfully restored and read back. | Require an owner-signed per-action restore manifest + rehearsal event before the teardown gate can advance. |
| F61 — retired-epoch transition | No server control proves every stale tab/service/retry was rejected between final team-zero/export and retired-mode activation. | Add a server maintenance/cutoff or atomic high-water transition; page on any accepted cross-boundary write and require exact zero before resume. |
| F63 — operator SQL executability | Prose review does not prove a paste fence is one action, valid SQL, compare-and-set safe, or transactionally exercised. | Parse/lint every fence and run every action against an isolated TEST flag store; publish the resulting action/hash/readback manifest. |

| F66 — shared alert-secret blast radius | Removing the shared project alert secret can silence both Linear-inbound anomalies and onboarding fallback notifications. | Add lane-specific controls/secrets and independent TEST heartbeats; never use shared-secret removal as scoped containment. |
| F67 — unauthenticated fallback writers | EF auth telemetry cannot see direct or routed writes to legacy n8n fallback webhooks. | Authenticate/scope or retire every fallback; run direct negative probes and page on any anonymous accepted write. |
| F68 — comment-import rollback dependencies | No current preflight detects cross-run self-FK descendants or post-import versions before deletion. | Keep rollback blocked; require a fresh dependency/version graph and assertion-bearing TEST/scratch rehearsal before documenting a command. |

## Alert destinations

- n8n workflow failures do **not** yet have blanket alert coverage (F09). The central
  `SyncView - Error Alerts -> DM Sidney` handler exists, but a 2026-07-14 live settings sample found
  five of six load-bearing cutover workflows unwired; three of those had 135 error/crash/cancel
  records since Jul 7. The handler itself also failed 29 of 30 sampled invocations at the execution
  limit. Treat silence as unknown until a machine census proves every load-bearing active workflow
  points to a tested handler and the independent non-n8n pager is live.
- The n8n execution-quota Action runs outside n8n. Threshold notices reuse the active Edge Alert Relay, while a failed n8n read or relay confirmation leaves a failed GitHub run rather than a false green or a consumed monthly marker. Secrets are stored as `N8N_API_KEY` and `N8N_QUOTA_ALERT_WEBHOOK`; the plan cap is the repository variable `N8N_MONTHLY_EXECUTION_CAP`.
- Edge Function anomaly alerts use the project `SLACK_ALERT_WEBHOOK` secret and send public-safe messages only: anomaly type, Linear issue identifier, and team. They do not include client names, assignee names, emails, or raw payloads. `onboarding-capture` also consumes this secret, so removing it is not a lane-specific rollback (F66). n8n workflow `Tfhc3vebZyG6obOg` is the active Edge Alert Relay to DM Sidney; synthetic relay execution `219237` and real EF anomaly relay execution `219602` succeeded on 2026-07-08.
- n8n workflow `qllIDZPkdNAPRj0b` is the active Monitoring Pager + Reconciler Trigger. It runs every 15 minutes, dispatches `linear-deliverables-reconcile.yml` with `apply=false`, dispatches the calendar and samples reconcilers with `dry_run=false` and cap 15, dispatches the B1 incremental refresh through a 30-minute gate, and actively dispatches `linear-outbound-drain.yml`. It reads each outbound summary; because normal mode is `off` and parity is separately disabled, outbound-summary staleness paging is suppressed. It throttles each DM alert condition to at most once per hour.
- The write-UI soak watcher is repository-staged only. After the scheduled workflows merge, qll adds two
  aggregate readers and alerts on stale/failed write drills, echo or reconcile counts, unexpected
  shadow work, or a failed zero-write proof. Alert text contains only team labels, aggregate counts,
  and event handles. It does not contain client/project names or payload bodies.

## Nightly test boundaries

- Calendar and Samples nightlies are scoped to the private active TEST fixture and dummy/test rows; each probe cleans up after itself.
- The Production preview smoke and polish gate are read-only. They serve `index.html`, open `?prod=1`, verify migrated rows render, assert zero write-like browser requests, and publish validated review artifacts for visual inspection.
- The production-write drill is service-authenticated, requires exactly one active `kind=test`
  client, and lets the gateway resolve its server-private TEST project allowlist. It introduces no
  new GitHub secrets. It uses the gateway's TEST override without changing
  `prod_authority` or `linear_outbound_enabled`, archives its disposable fixtures, and uploads only
  a public-safe aggregate. The shadow audit reads real-client rows but makes zero Linear, flag,
  outbox, or authoritative-row mutations; its row-level file stays in runner-temporary storage and
  is never uploaded.
